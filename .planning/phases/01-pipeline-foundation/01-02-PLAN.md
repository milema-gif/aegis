    1. Create `skills/aegis-launch.md` — Claude Code command file with YAML frontmatter:

       The command body should:
       - State the objective: "Launch or resume the Aegis pipeline for a project."
       - Reference the orchestrator: "@workflows/pipeline/orchestrator.md"
       - Pass $ARGUMENTS as the project name
       - Keep the command file LEAN (under 30 lines). All logic lives in the orchestrator.

    2. Create `workflows/pipeline/orchestrator.md` — the core orchestration workflow:

       This is the brain of Aegis. It is a prompt/workflow document (not a script) that Claude follows when dispatched from `/aegis:launch`. Structure it as a step-by-step process:

       **Step 1 — Resolve project:**
       - If $ARGUMENTS provided, use as project name
       - If no arguments, look for .aegis/state.current.json in current directory
       - If neither, ask user for project name

       **Step 2 — Load or initialize state:**
       - Check if .aegis/state.current.json exists
       - If yes: read it, validate JSON (if corrupt, run recover_state logic per lib/aegis-state.sh)
       - If no: initialize new pipeline (create .aegis/ directory, copy templates/pipeline-state.json, replace placeholders, set intake as active)

       **Step 3 — Detect integrations (every invocation, not cached):**
       - Probe for Engram: attempt a lightweight call (or check for markers)
       - Probe for Sparrow: check if /home/ai/scripts/sparrow exists and is executable
       - Update state.current.json integrations section
       - Note: Codex availability mirrors Sparrow (same script, --codex flag). Always gated: "user-explicit only"

       **Step 4 — Announce pipeline status:**
       - Display the formatted banner (per references/integration-probes.md):
         ```
         === Aegis Pipeline ===
         Project: {name}
         Stage: {stage} ({index+1}/9)

         Integrations:
           [OK] or [MISSING] for each integration
         ```

       **Step 5 — Dispatch to current stage:**
       - Read current_stage from state
       - Look for workflow file at workflows/stages/{stage-name}.md
       - If workflow exists: follow it
       - If workflow missing (Phase 1 stubs): announce "Stage '{name}' workflow not yet implemented. Auto-completing for pipeline demonstration."
       - After stage completes: advance to next stage (using transition logic from lib/aegis-state.sh)
       - If current stage is "advance": check roadmap for remaining phases, pass count to advance_stage

       **Step 6 — Post-transition:**
       - Journal the transition
       - Update state file atomically
       - If more stages remain and auto_advance is true: loop back to Step 4
       - If auto_advance is false: announce next stage and wait for user to re-invoke

       **Important orchestrator rules (embed in the workflow):**
       - NEVER accumulate stage outputs in conversation context. Write to files, reference by path.
       - ALWAYS read state from file at start of each invocation. Never trust in-memory state.
       - Use the transition TABLE, not if/else chains, for stage ordering.
       - Detect integrations EVERY invocation (cheap probes, stale cache is worse than re-probing).
  </action>
