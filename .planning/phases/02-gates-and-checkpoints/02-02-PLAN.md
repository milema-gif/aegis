  <action>
**Part A: Update `workflows/pipeline/orchestrator.md`**

Add `lib/aegis-gates.sh` to the Libraries section at the top alongside the existing three libraries. Description: "Gate evaluation: evaluate gates, display banners/checkpoints, track retries".

Insert a new **Step 5.5 -- Evaluate Gate** between Step 5 (Dispatch) and Step 6 (Post-Transition). This step replaces the direct advance_stage() call that currently lives at the bottom of Step 5. The new flow:

**Step 5 changes:** Remove the `advance_stage()` call from the bottom of Step 5. Step 5 now ONLY dispatches to the stage workflow. When the stage workflow signals completion, control falls through to Step 5.5.

**New Step 5.5 -- Evaluate Gate:**

```
After a stage signals completion, evaluate its gate before advancing:

1. Source lib/aegis-gates.sh
2. Read current stage and YOLO mode from state
3. Call init_gate_state(current_stage) to set first_attempt_at
4. Call check_gate_limits(current_stage):
   - If "retries-exhausted": show error box, set stage status to "failed", STOP
   - If "timed-out": show error box, set stage status to "failed", STOP
   - If "ok": proceed to gate evaluation
5. Call evaluate_gate(current_stage, yolo_mode):
   - "pass": show_transition_banner(), proceed to advance_stage()
   - "auto-approved": show_yolo_banner(), show_transition_banner(), proceed to advance_stage()
   - "approval-needed": show_transition_banner(), show_checkpoint(), set_pending_approval(stage, true), STOP (do not advance, do not auto-advance)
   - "fail": record_gate_attempt(stage, "fail", error), show retry banner with attempts remaining, STOP (do not advance)
6. If gate passed: call advance_stage() (transitions to next stage)
```

**Step 2 changes:** Add a sub-step after loading state: check if any stage has `pending_approval: true`. If found:
- Show: "Pending approval for stage '{name}'."
- Present the checkpoint box again
- Wait for user input
- On "approved": clear pending_approval via set_pending_approval(stage, false), then call advance_stage() and continue to Step 4
- On rejection: remain at current stage, proceed to Step 4 (which will re-announce current stage)

**Step 6 changes:** Update the auto-advance loop. Add this condition: "If the gate result from Step 5.5 was 'approval-needed' or 'fail', do NOT auto-advance regardless of config.auto_advance. Auto-advance only proceeds when the gate result was 'pass' or 'auto-approved'."

**Add to Important Rules:** Rule 5: "Gate evaluation happens ONCE per stage completion attempt. If a gate fails, the stage must be re-executed (re-invoke /aegis:launch) before the gate is re-evaluated. Do NOT retry the gate in-conversation -- this burns context."

**Add to Handled Scenarios table:**
| Gate blocks advance | Stage re-enters active, user re-invokes to retry |
| Approval gate pauses | Checkpoint displayed, pending_approval set, pipeline stops |
| YOLO auto-approval | Approval gate auto-approved with compact banner, quality gates still enforced |
| Gate retries exhausted | Stage set to "failed", error displayed, user must intervene |
| Gate timeout | Stage set to "failed" based on wall-clock, user must intervene |
| Pending approval on resume | Checkpoint re-displayed at Step 2, user can approve or reject |

**Part B: Update `lib/aegis-state.sh`**

Add a `source` guard at the top of aegis-gates.sh sourcing: do NOT modify aegis-state.sh to source aegis-gates.sh (avoid circular dependency). Instead, add a helper function `read_yolo_mode()` to aegis-state.sh that reads config.yolo_mode from state.current.json and prints "true" or "false". This is a simple state-read function that belongs in the state library.

Also add `read_stage_status(stage_name)` to aegis-state.sh that reads a specific stage's status field and prints it. This is used by evaluate_gate to check if a quality gate's stage is completed.

Both functions use python3 for JSON reading (Phase 1 convention).
